# [PERF] Image Optimization

## 이슈 개요
메인페이지 진입 시 KOPIS API 이미지 로딩 속도가 느려 사용자 경험 저하

## 현재 상태 (AS-IS)
- KOPIS API에서 제공하는 외부 이미지 URL을 직접 참조
- 예시: `http://www.kopis.or.kr/upload/pfmPoster/PF_PF275514_251002_105226.jpg`
- 매 요청마다 KOPIS 서버에서 이미지를 직접 다운로드
- 메인 페이지 렌더링 시 여러 이미지가 동시에 로드되면서 지연 발생
- 캐싱 메커니즘 없음

### 문제점
1. **느린 로딩 속도**: 외부 서버 의존으로 인한 네트워크 지연
2. **안정성 이슈**: KOPIS 서버 상태에 따라 이미지 로드 실패 가능
3. **대역폭 낭비**: 동일 이미지 반복 다운로드
4. **사용자 경험 저하**: 이미지 로딩 대기 시간 증가

## 목표 상태 (TO-BE)
- 백엔드에서 KOPIS 이미지를 프록시/캐싱하여 제공
- 이미지 캐싱 전략 도입 (Redis 또는 디스크 캐시)
- 필요시 이미지 리사이징 및 WebP 변환으로 용량 최적화
- 프론트엔드에서는 로컬 서버 URL로 이미지 요청
- 예시: `http://localhost:8081/api/v1/images/proxy/{imageId}`

### 기대효과
1. **빠른 로딩 속도**: 캐싱된 이미지 즉시 제공
2. **안정성 향상**: 서버 내부 캐시로 KOPIS 서버 장애 대응
3. **대역폭 절감**: 한 번 다운로드한 이미지 재사용
4. **사용자 경험 개선**: 매끄러운 페이지 로딩

## 구현 방안

### Option 1: 이미지 프록시 + 캐싱 (권장)
- Spring Boot에서 이미지 프록시 컨트롤러 추가
- Redis 또는 디스크 기반 캐싱 구현
- 캐시 만료 정책 설정 (예: 7일)

### Option 2: 이미지 사전 다운로드
- KOPIS API 동기화 시 이미지 함께 다운로드
- 로컬 스토리지에 WebP 형식으로 저장
- DB에 로컬 파일 경로 저장

### Option 3: CDN 활용
- Cloudflare Images 또는 Vercel Image Optimization
- 외부 서비스 비용 발생 가능
